## Последовательность действий

1. Оценить состояние версии проекта по файлу .version:

  - Данный файл должен находиться в корне проекта;
  - Если он не был изменён с момента прошлого коммита, то это означает, что версию следует обновить в соответствии с инструкциями ниже;

1.1. Оценить изменения, подлежащие коммиту:

  - Изменения включают в себя: изменённые (modified), новые (untracked) и удалённые файлы;
  - Если изменений нет — сообщить об этом пользователю и завершить выполнение скилла;

2. Принять задачу от пользователя с указанием того, какая версия проекта должна быть закоммичена и запушена:

  - Если явного указания нет, то версия инкрементируется на уровне патча, и, если файл .version не обновлялся с момента последнего коммита, то указанным образом инкрементируется именно 2-я строка этого файла, после чего вызывается команда `make update-version` для обновления версии в файле .version:

    - Инкрементация 2-й строки файла .version осуществляется через непосредственную модификацию файла;
    - Вызов команд `make ...` применительно к функционалу проекта makefile должен производиться в терминале, находясь в корне проекта;
    - Убедитесь в том, что после вызова команды `make update-version` файл .version был обновлён, и его 1-я и 2-я строка совпадают;

3. Делается соответствующая запись в `CHANGELOG.md`, при этом:

  - Последнее изменение записывается на самом верху файла;
  - Указывается дата коммита в формате YYYY-MM-DD;
  - Указывается версия проекта, которая была получена в результате изменения версии на предыдущем шаге;
  - Под изменениями следуют подзаголовки: Added, Changed, Fixed, Deprecated, Removed, Security;
  - Ниже каждого из подзаголовков перечисляются изменения, относящиеся к этому подзаголовку, при этом максимально кратко;
  - Самое значимое изменение следует разместить первым — на его основании будет создано сообщение коммита;
  - Изменения в CHANGELOG.md делаются только на английском языке;

4. Если с прошлого коммита файл README.md не изменялся, то пользователю предлагается отразить изменения в файле README.md, и если данное действие подтверждается пользователем, то:

  - Файл дополняется на том языке, на котором он был представлен изначально;

5. Финальным шагом пользователю предлагается текст команды `make git_commit_push "<message>"`, где `<message>` - это сообщение коммита, которое:

  - Должно быть максимально кратким и отражать самое главное изменение, идущее первым в списке изменений в CHANGELOG.md;
  - Если в CHANGELOG.md допустимо более развёрнутое описание, то в сообщении коммита следует быть максимально кратким, так как оно будет отображаться в истории коммитов и должно быть легко читаемым;
  - Версия проекта не указывается в сообщении коммита — она добавляется автоматически;

5.1. Запускать эту команду пользователь должен сам!
